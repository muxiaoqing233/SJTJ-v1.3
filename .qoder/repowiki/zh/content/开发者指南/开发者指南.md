## 环境配置

本项目采用前后端分离架构，后端基于Node.js + Express + SQLite技术栈，前端使用原生JavaScript和D3.js进行可视化。开发者需按照以下步骤配置开发环境。

### 开发环境要求
- **Node.js**: 14.0 或更高版本
- **npm**: 6.0 或更高版本
- **SQLite3**: 用于数据库存储
- **文本编辑器**: 推荐使用支持ESLint和Prettier的编辑器（如VS Code）

### 环境变量配置
项目通过`.env`文件管理环境变量，位于`backend/.env`。关键配置如下：
- `PORT=3001`：后端服务监听端口
- `NODE_ENV=development`：运行环境（开发/生产）
- `JWT_SECRET=your-super-secret-jwt-key-here`：JWT令牌密钥（生产环境必须修改）
- `DB_PATH=./data/military-knowledge.db`：SQLite数据库文件路径

开发环境建议使用默认配置，生产环境应修改`JWT_SECRET`为至少32位的随机字符串，并调整日志级别为`warn`。

### 项目启动
```bash
# 进入后端目录
cd backend

# 安装依赖（首次运行）
npm install

# 启动开发服务器
npm run dev
```
服务器将在 `http://localhost:3001` 启动。前端页面可通过浏览器访问根目录下的HTML文件。

## 代码规范与提交

### 代码风格
- **JavaScript**: 遵循ES6+语法，使用`const`和`let`代替`var`
- **命名规范**: 采用驼峰命名法（camelCase）
- **缩进**: 使用2个空格
- **分号**: 语句结尾必须加分号
- **字符串**: 优先使用单引号

### 提交规范
代码提交应遵循以下规范：
1. **原子性提交**：每次提交只包含一个功能或修复
2. **清晰的提交信息**：使用“动词+描述”格式，如`fix: 修复武器图片ID转换问题`
3. **关联问题**：若修复特定问题，应在提交信息中引用问题编号

### 代码质量工具
项目已集成以下工具确保代码质量：
- **ESLint**: 代码质量检查
- **Prettier**: 代码格式化
- **Nodemon**: 开发热重载

## 调试技巧

### 常见问题排查
通过分析`function_description`目录中的修复说明，总结出以下常见问题排查思路：

#### 1. 武器图片功能无法显示
**问题诊断**：
- 数据库中`weapons`表的`images`字段为空
- 前端ID格式与后端不匹配（`weapon_2` vs `2`）
- API路径未使用完整URL

**解决方案**：
- 检查`backend/src/routes/weapon-images.js`中的ID转换逻辑
- 确保前端使用完整API路径`http://localhost:3001/api/weapon-images/${weaponId}`
- 在`scripts/weapon-image-modal.js`中添加错误处理和重试机制

#### 2. 制造商API无法获取数据
**问题诊断**：
- 武器API未关联制造商表
- 前端只尝试连接3001端口，但服务器可能运行在3000端口

**解决方案**：
- 修改`backend/src/routes/weapons-simple.js`，使用`LEFT JOIN`关联`manufacturers`表
- 在`scripts/knowledge-graph-analysis-manufacturer-complete.js`中添加多端口尝试连接机制

### 日志分析
系统使用Winston日志库，日志文件位于`backend/logs/app.log`。可通过以下命令查看实时日志：

## 分支管理

### 分支策略
项目采用简单的分支管理策略：
- `main`：主分支，存放稳定版本代码
- `develop`：开发分支，集成所有功能开发
- `feature/*`：功能分支，用于开发新功能
- `hotfix/*`：紧急修复分支

### 工作流程
1. 从`develop`分支创建功能分支：`git checkout -b feature/new-feature develop`
2. 在功能分支上进行开发和提交
3. 开发完成后，合并到`develop`分支：`git checkout develop && git merge feature/new-feature`
4. 定期将`develop`分支合并到`main`分支发布新版本

### 发布流程
1. 确保`develop`分支通过所有测试
2. 合并`develop`到`main`分支
3. 为`main`分支打版本标签：`git tag -a v1.3 -m "Release version 1.3"`
4. 推送标签到远程仓库：`git push origin v1.3`
